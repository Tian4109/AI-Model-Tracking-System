# Project Workflow

## Purpose

This project automatically collects current large language model information from selected vendor sources and generates a structured comparison report for client reference and model selection.

The workflow supports:

- official-source-first research
- structured data extraction
- multi-vendor comparison
- AI-assisted recommendations
- repeatable documentation generation

---

# High-Level Workflow

```text
Official Links
        ↓
Claude Code Command
        ↓
Vendor Researcher Agent
        ↓
Structured Draft Output
        ↓
Reporter Agent
        ↓
Final Report (models.md)
```

---

# Core Components

| Component | Responsibility |
|---|---|
| `links.md` | Official and official-ecosystem source list |
| `.claude/commands/` | User-facing workflow entry points |
| `BaseResearcher.md` | Shared research and extraction rules |
| `Researcher-*.md` | Vendor-specific research configuration |
| `Researcher-Vendor.md` | Generic Researcher for dynamically added vendors |
| `drafts/*.md` | Structured intermediate outputs |
| `Reporter.md` | Final synthesis and recommendations |
| `models.md` | Final client-facing report |

---

# Research Workflow

Each vendor Researcher follows this process:

```text
Read links.md
        ↓
Open official vendor sources
        ↓
Use supplemental official search if needed
        ↓
Extract structured model information
        ↓
Normalise fields
        ↓
Generate vendor draft
```

Researcher agents:

- prioritise official sources
- avoid unsupported guesses
- standardise pricing and context formatting
- separate API and open-weight models
- preserve source traceability

---

# Reporting Workflow

The Reporter agent:

```text
Read all drafts
        ↓
Compare vendors
        ↓
Normalise terminology
        ↓
Generate recommendations
        ↓
Write models.md
```

The Reporter does not browse the web.

It only uses structured draft outputs generated by the Researcher agents.

---

# Supported Workflows

## 1. Vendor-Specific Update

Use when only one vendor needs updating.

Example:

```text
/track-openai
/generate-report
```

This workflow:

- updates one vendor draft
- rebuilds the final report
- reduces runtime and token usage

---

## 2. Generic Vendor Update

Use when a vendor is added to `links.md` but does not have a dedicated command.

```text
/track-vendor <VendorName>
/generate-report
```

Example:

```text
/track-vendor Mistral
/generate-report
```

This workflow:

- reads the matching vendor section from `links.md`
- uses `Researcher-Vendor`
- generates `drafts/<vendor>.md`
- includes the vendor in `models.md` through `/generate-report`

---

## 3. Full Refresh Workflow

Use when all vendors may have changed.

```text
/update-report
```

This workflow:

- refreshes all drafts
- regenerates the final report
- takes longer but ensures full consistency

---

## 4. Draft-Only Refresh

```text
/track-all
```

This updates all core vendor drafts and additional vendor drafts without generating the final report.

Useful for staged review workflows.

---

# Draft File Role

Files inside `drafts/` are structured intermediate outputs.

They are not intended to be the final client-facing deliverable.

Normally:

- Researcher agents generate them
- Reporter reads them
- users do not manually edit them

---

# Recommendation Philosophy

The system intentionally uses AI-assisted reasoning rather than a hard-coded rules engine.

Recommendations are based on:

- pricing
- context window
- modality support
- deployment suitability
- licensing
- practical strengths

The report aims to provide practical guidance rather than rigid rankings.

---

# Source Policy

The project follows an official-source-first policy.

Allowed sources include:

- official vendor documentation
- official pricing pages
- official GitHub repositories
- official Hugging Face organisation pages
- official model cards
- official announcements

Unofficial blogs, random comparison pages, and community summaries should not be treated as authoritative sources.

---

# Output Philosophy

The final report should be:

- concise
- structured
- decision-oriented
- readable for both technical and non-technical stakeholders

The goal is not to create a benchmark leaderboard.

The goal is to help answer:

```text
Which model should we consider for this situation, and why?
```


---

# Generic Vendor Discovery in `/track-all`

`/track-all` treats `links.md` as the vendor registry.

It should:

1. Update the five core vendors with their dedicated Researcher agents.
2. Scan `links.md` for any additional top-level vendor sections.
3. Use `Researcher-Vendor` for additional vendors.
4. Generate matching files in `drafts/`.

This keeps the system extensible while preserving higher-quality dedicated Researchers for the core vendors.
