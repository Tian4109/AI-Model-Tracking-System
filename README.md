# AI Model Tracking Documentation System

## Project Overview

This project is a Claude Code based AI model tracking and documentation system.

It automatically collects current large language model information from selected vendor sources, writes structured vendor draft files, and generates a consolidated Markdown report for model comparison and selection.

The final client-facing output is:

```text
models.md
```

The system is designed for client-side reference use. It helps users compare major model providers and choose suitable models for different scenarios such as coding, reasoning, long-context processing, multimodal tasks, local deployment, and cost-sensitive workloads.

---

## What This Project Produces

| Output | Purpose |
|---|---|
| `drafts/*.md` | Intermediate structured vendor research outputs |
| `models.md` | Final consolidated model comparison and recommendation report |

The `drafts/` files are generated by Researcher agents.

The `models.md` file is generated by the Reporter agent.

---

## Supported Vendors

Current core vendors:

- OpenAI
- Anthropic
- Google
- Meta
- Microsoft

Additional vendors can be added through `links.md` and updated with `/track-vendor <VendorName>`.

The project covers both:

- API / closed-weight models
- Open-weight models

Chinese-origin model families are intentionally excluded from the current project scope unless the scope is changed later.

# Quick Start for New Users

This section explains how a new user can run the project from zero.

---

## 1. Install Claude Code

Claude Code is the command-line tool used to run this project.

Install Claude Code using the official Claude Code installation instructions for your operating system.

For Windows PowerShell:

```powershell
irm https://claude.ai/install.ps1 | iex
```

For Windows CMD:

```cmd
curl -fsSL https://claude.ai/install.cmd -o install.cmd && install.cmd && del install.cmd
```

For macOS / Linux / WSL:

```bash
curl -fsSL https://claude.ai/install.sh | bash
```

After installation, check that Claude Code is available:

```bash
claude --version
```

If the command is not recognised, restart the terminal or check whether the Claude Code installation path has been added to the system PATH.

---

## 2. Sign In to Claude Code

Open a terminal and run:

```bash
claude
```

Claude Code should guide you through authentication.

Depending on your access method, you may sign in with:

- a Claude subscription account
- an Anthropic Console/API account
- an organisation-managed account

Follow the browser or terminal prompts until Claude Code is authenticated.

---

## 3. Open the Project Folder

Open this project folder in VS Code or another code editor.

Recommended for beginners:

```text
Open VS Code
→ File
→ Open Folder
→ Select this project folder
```

Then open the integrated terminal:

```text
Terminal
→ New Terminal
```

Make sure the terminal is inside the project root folder.

The project root should contain:

```text
README.md
links.md
WORKFLOW.md
.claude/
drafts/
```

You can check the current folder by running:

```bash
pwd
```

or on Windows CMD:

```cmd
cd
```

---

## 4. Start Claude Code in the Project Folder

In the project root terminal, run:

```bash
claude
```

Claude Code should start inside this project.

This is important because Claude Code needs to detect:

```text
.claude/commands/
.claude/subagents/
links.md
drafts/
```

Do not start Claude Code from the wrong directory.

---

## 5. Run a Test Command

For the first test, use a single-vendor command instead of a full update.

Recommended first test:

```text
/track-openai
```

Expected result:

```text
drafts/openai.md
```

Then generate the final report:

```text
/generate-report
```

Expected result:

```text
models.md
```

---

## 6. Full Update Command

Use this only when all vendors need to be refreshed:

```text
/update-report
```

This command may take longer because it updates all vendor drafts and then generates the final report.

---

## 7. Review the Output

After running commands, review:

```text
drafts/

```

Check that:

- vendor drafts were created or updated
- `models.md` was generated
- model information appears structured
- recommendations are reasonable
- missing values are shown as `—`

# Common Commands

## Targeted Vendor Update

Use this when only one vendor needs updating.

```text
/track-openai
/generate-report
```

Other vendor commands:

```text
/track-anthropic
/track-google
/track-meta
/track-microsoft
```

---

## Generic Vendor Update

Use this when a vendor has been added to `links.md` but does not have a dedicated Researcher command.

Example:

```text
/track-vendor Mistral
/generate-report
```

Requirements:

- The vendor must have a top-level section in `links.md`, for example `# Mistral`.
- The vendor name used in the command must match the section title.
- `Researcher-Vendor.md` will generate a matching draft file such as `drafts/mistral.md`.
- `/generate-report` will include all available `drafts/*.md` files in `models.md`.

This allows users to add new vendors mainly by maintaining `links.md`, without creating new command or subagent files.

---

## Full Draft Refresh

Use this when all core vendor drafts and additional vendor drafts should be updated, but the final report will be generated separately.

```text
/track-all
```

Then run:

```text
/generate-report
```

---

## Full End-to-End Refresh

Use this when most or all vendor information may be outdated.

```text
/update-report
```

---

# Command Decision Guide

| Need | Recommended command | Output |
|---|---|---|
| Update one core vendor only | `/track-openai`, `/track-google`, etc. | One `drafts/*.md` file |
| Update one additional vendor | `/track-vendor <VendorName>` | One `drafts/*.md` file |
| Rebuild final report from existing drafts | `/generate-report` | `models.md` |
| Refresh all vendor drafts only | `/track-all` | All core and additional `drafts/*.md` files |
| Full end-to-end refresh | `/update-report` | All drafts + `models.md` |

## Main Components

| Component | Role |
|---|---|
| `links.md` | Official source list and source management file |
| `.claude/commands/` | User-facing Claude Code command entry points |
| `BaseResearcher.md` | Shared research rules and output standard for all Researcher agents |
| `Researcher-*.md` | Vendor-specific research configuration and special notes |
| `Researcher-Vendor.md` | Generic Researcher for vendors added dynamically through `links.md` |
| `drafts/*.md` | Structured intermediate vendor outputs |
| `Reporter.md` | Final synthesis, comparison, recommendation, and report generation agent |
| `models.md` | Final client-facing model reference report |

# Source Policy

This project follows an official-source-first policy.

- `links.md` is the primary source list.
- Researcher agents should use official vendor documentation first.
- Supplemental search is allowed when it leads to strong official or official-ecosystem sources.
- Suitable sources include official documentation, pricing pages, model cards, official GitHub repositories, official Hugging Face organisation pages, and official announcements.
- Third-party blogs, Reddit, unofficial benchmark sites, and random comparison pages should not be used as final evidence.
- Missing or unclear values should be marked as `—`, not guessed.

# Researcher Design

The project uses a shared-template plus vendor-specific configuration approach.

## `BaseResearcher.md`

`BaseResearcher.md` defines common rules for all Researcher agents, including:

- official-source-first research workflow
- required data fields
- open-weight vs open-source classification
- pricing format
- context length format
- missing information handling
- draft output structure
- quality checklist

## Vendor-specific Researcher files

Each vendor Researcher file provides:

- vendor name
- output draft path
- relevant `links.md` section
- vendor-specific model families
- vendor-specific classification notes
- special warnings or exclusions

This design keeps vendor-specific control while avoiding repeated prompt logic across multiple Researcher files.

---

# Draft Files

Files inside `drafts/` are generated intermediate outputs.

They should be treated as structured data sources for the final report, not as final user-facing documentation.

Normally, they should not be manually edited.

If model information is wrong or outdated:

```text
1. Update the relevant source links or Researcher instructions
2. Run the matching /track-* command
3. Run /generate-report
```

---

# Reporter Design

`Reporter.md` is not a simple Markdown merger.

It should:

- read all vendor draft files
- compare models across vendors
- organise API and open-weight models clearly
- summarise pricing, context length, modality, licensing, and deployment options
- generate scenario-based recommendations
- explain recommendation reasoning
- produce the final client-facing `models.md`

The final report should help users make model selection decisions, not merely list raw model data.

---

# Final Report Expectations

The final `models.md` should include:

- generation date
- source scope
- introduction
- API / closed-weight model section
- open-weight model section
- comparative analysis
- pricing comparison
- context window comparison
- multimodal capability comparison
- scenario-based recommendations
- summary and key trends
- reminder to verify current vendor pricing before procurement decisions

# Model Classification Notes

## API / Closed-weight models

These models are accessed mainly through hosted APIs and do not provide public downloadable weights.

Examples:

- OpenAI GPT models
- Anthropic Claude models
- Google Gemini API models

## Open-weight models

These models provide publicly available weights for download or local deployment.

Examples:

- Meta Llama
- Google Gemma
- Microsoft Phi
- OpenAI GPT-OSS

## Open-weight vs open-source

Open-weight does not always mean open-source.

A model should only be treated as open-source if the official source clearly states a recognised open-source licence such as Apache 2.0 or MIT.

Community licences and custom vendor licences should be described accurately rather than automatically called open-source.

# Troubleshooting

## Claude Code command not found

Try:

```bash
claude --version
```

If it fails:

- restart the terminal
- check Claude Code installation
- check system PATH

## Custom slash commands do not appear

Make sure Claude Code was started from the project root folder.

The project root must contain:

```text
.claude/commands/
```

## Draft file was not generated

Check:

- Claude Code has permission to write files
- the `drafts/` folder exists
- the command was run from the project root
- the correct `/track-*` command was used
- for `/track-vendor <VendorName>`, the vendor section exists in `links.md` and the name matches the section title

## `models.md` was not generated

Check:

- at least one draft file exists in `drafts/`
- `/generate-report` was run
- `Reporter.md` exists in `.claude/subagents/`

## Output looks outdated

Update `links.md` if needed, then rerun the relevant command.

For one vendor:

```text
/track-openai
/generate-report
```

For all vendors:

```text
/update-report
```
